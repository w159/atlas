# Project Glossary

**In brief:** A single, widely accessible place that defines all the specific terms and vocabulary used on a project, so everyone uses the same words for the same things.

**Category:** Practice
**Source:** Chapter 8, Topic 45 "The Requirements Pit"
**Also known as:** none

## What it is
As soon as you start discussing requirements, users and domain experts use terms that carry specific meaning to them. They may distinguish a "client" from a "customer," for example, and it would be wrong to use either word casually in the system.

A project glossary is one place that defines all the specialized terms and vocabulary used in a project. Every participant, from end users to support staff, uses it to keep language consistent. Because everyone needs it, the glossary should be widely accessible, which is a strong argument for keeping it as online documentation.

## Why it matters
It is hard to succeed when users and developers call the same thing by different names, or, worse, refer to different things by the same name. That kind of vocabulary drift creates miscommunication that surfaces as defects, rework, and mismatched expectations between the people who need the system and the people building it.

## In practice
- Create the glossary as soon as requirements discussions begin, and maintain it as new terms appear.
- Capture the precise domain distinctions users make (client vs. customer, and so on).
- Keep it online and widely accessible so end users, developers, and support staff all reference the same source.
- Treat it as the canonical naming authority for the system's vocabulary.
