# Live MCP Server Test - 2026-06-12 (after extension updates)

Tested the connected (installed) MCP extensions by calling real tools. The most important
finding first: the connected servers are PUBLISHED extension versions, not the local `.mcpb`
files rebuilt earlier in this repo. Claude Desktop "check for updates" pulls published builds;
it does not install the local `mcp_servers/*/*.mcpb` files. Evidence: ThreatLocker still has the
bug the repo build fixes, KnowBe4 is navigation-gated (the repo build lists all tools directly),
and ConnectWise `cw_search_tickets` still has the old signature without `fields`/`full`.

## Fix made this session

`mcp_servers/_shared/base-url.ts` now rejects three failure modes for the optional base-URL env
var: blank, an unexpanded `${user_config.*}` manifest placeholder (observed on Paylocity), and
any non-http value - all fall back to the documented vendor default. This hardens the five
servers that use the shared resolver for live calls (auvik, knowbe4, ninjaone, kaseya-spanning,
paylocity) against a silently-broken base URL when the optional field is left blank. The five
bundles should be rebuilt to ship it.

## The key takeaway

Updating extensions from the marketplace pulled published builds that differ from the repo's
rebuilt bundles. The repo's fixes (ThreatLocker runtime methods, ConnectWise shaping, KnowBe4
paths, etc.) reach the running environment only when the local `.mcpb` files in `mcp_servers/`
are installed directly into Claude Desktop - not via "check for updates."
